# 최종실행명세 통합 2 최종본

## 1) 프로젝트 개요

본 프로젝트는 이미 입력된 구글 시트의 족보 데이터를 기반으로, 모바일 반응형 PWA 족보 앱을 구축하는 것을 목표로 한다.

기본 구조는 다음과 같다.

**Google Sheets → Apps Script(JSON API) → PWA 프론트엔드 → GitHub/Netlify 배포**

이 구조에서:

- 구글 시트는 원본 데이터 저장소
- Apps Script는 데이터 가공 및 JSON 제공 API
- PWA 프론트엔드는 실제 사용자 화면
- GitHub는 버전 관리
- Netlify는 정적 프런트엔드 배포

로 역할을 분리한다.

앱의 제목과 부제는 다음과 같이 한다.

- **제목:** 의성김씨 아천문중 세보
- **부제:** 디지털 족보 및 문중 정보 시스템

홈 화면의 기본 배경 톤은 연한 하늘색으로 한다.

## 2) 개발 원칙

### 2-1. 데이터와 화면 분리
데이터는 구글 시트와 Apps Script에서 관리하고, 화면은 Netlify에 배포되는 정적 프런트엔드로 분리한다.

### 2-2. 모바일 우선
앱은 데스크톱보다 모바일 사용성을 우선으로 설계한다.

### 2-3. PWA 적용
다음 요소를 적용한다.

- `manifest.json`
- `service worker`
- app shell 구조
- 정적 자원 캐시

단, 데이터는 최신 반영을 위해 **network-first** 전략을 사용한다.

### 2-4. 성능 최적화
초기 데이터 로딩 후 메모리 인덱스를 생성한다.

필수 인덱스:

- `peopleById`
- `childrenByFather`
- `searchIndex`
- `peopleByGeneration`

필요 시 추가:

- `siblingsByFather`
- `peopleByName`
- `spousesByPerson`

## 3) 전체 페이지 구성

앱은 최소 다음 4개 영역으로 구성한다.

1. **홈**
2. **가계도**
3. **선조의 발자취**
4. **아천문중**

하단에는 항상 플로팅 내비게이션을 배치한다.

내비게이션 버튼:

아이콘과 텍스트를 함께 표시한다.

## 4) 홈 화면 명세

홈 화면은 다음 순서로 구성한다.

### 4-1. 상단 헤더
상단 헤더는 연한 하늘색 배경을 사용한다.

- **제목:** 의성김씨 아천문중 세보
- **부제:** 디지털 족보 및 문중 정보 시스템

### 4-2. 본인 확인 영역
상단 또는 하단에서 올라오는 Bottom Sheet 방식을 기본으로 한다.

구성:
- 이름 입력창
- 검색 버튼 또는 입력 즉시 검색
- 동명이인 후보 목록
- 후보 선택 후 본인 확정

본인 확인 영역은 사용자가 자신의 기준 인물을 확정하여, 이후 직계 요약, 세보 트리, 8촌 찾기, 촌수 계산 등의 기본 탐색 기준으로 삼기 위한 기능이다.

확정된 ID는 앱 상태에 저장되며, 필요 시 브라우저 저장 기능을 이용하여 재방문 시에도 유지할 수 있다.

단, 본인 확인은 본인 기준 탐색을 위한 기능일 뿐이며, 앱 전체에서는 **다른 문중원의 자료와 정보도 검색하여 함께 조회할 수 있도록** 설계한다.

### 4-3. 문중 알림판
구글 시트의 알림 시트와 연결하여 최신 공지 3개를 카드로 표시한다.

표시 항목:
- 작성일
- 제목
- 내용 요약
- 작성자

“더 보기” 버튼으로 전체 알림으로 이동 가능하게 확장할 수 있다.

### 4-4. 직계 요약 미리보기
검색된 본인의 직계 조상 일부를 홈 하단에서 미리 보여준다.

기본은 요약형으로 다음과 같이 표시한다.

- 본인
- 부 / 모
- 조부 / 조모
- 증조부 / 증조모
- 고조부 / 고조모

이 미리보기는 사용자가 자신의 직계 흐름을 한눈에 이해할 수 있도록 하는 간단한 요약 카드 형식으로 구성한다.  
모, 조모, 증조모, 고조모의 이름은 해당 직계 조상의 배우자 정보란에 기록된 이름을 기준으로 함께 표시한다.

## 5) 본인 확인 기능 명세

### 5-1. 입력 방식
사용자가 이름을 입력한다.

### 5-2. 검색 방식
`searchIndex`를 사용하여 이름으로 후보를 찾는다.

### 5-3. 동명이인 처리
동명이인이 존재하면 최대 10명까지 후보를 표시한다.

후보 표시 항목:
- 이름
- 아버지 이름
- 세손
- 필요 시 배우자 또는 출생연도

예:
- 김OO / 부: 김△△ / 28세
- 김OO / 부: 김□□ / 29세

### 5-4. 본인 확정
후보를 선택하면 해당 인물의 `id`를 `selectedPersonId`로 저장한다.

### 5-5. 본인 카드 표시
본인 확정 후 카드 형태로 다음 정보를 표시한다.

- 문중원 ID
- 이름
- 세손
- 배우자 이름
- 후손 표시
- 부 / 모
- 조부 / 조모
- 증조부 / 증조모
- 고조부 / 고조모

표시 방식은 1차로 카드 + 펼치기 구조를 권장한다.

모, 조모, 증조모, 고조모의 이름은 해당 직계 조상의 배우자 정보란에 기록된 이름을 기준으로 함께 표시한다.

후손 표시는 성별 및 데이터 구조에 따라 다음과 같이 처리한다.

- **남성 문중원**의 경우: 자녀 정보를 기준으로 표시한다.
- **여성 문중원**의 경우: 배우자란은 남편 이름으로 해석하고, 후손 표시는 외손란을 기준으로 표시한다.

여성 문중원의 외손 표시는 기본적으로 **최대 2명까지 노출**하며, 그 수를 초과할 경우에는 **“외 n명”** 형식으로 축약하여 표시한다.

예:
- 외손: 김○○, 박○○, 외 3명

자녀 또는 외손 정보가 많을 경우, 기본 일부만 보여주고 나머지는 “더 보기” 또는 축약 표시로 처리할 수 있다.

## 6) 직계 조상 연결표 명세

검색한 인물의 직계 조상을 위로 최대 30대까지 추적한다.

이 연결표에서는 각 세대의 직계 조상을 중심으로, 같은 아버지를 둔 형제를 함께 표시한다. 즉, 배우자 정보보다는 **부계 형제 관계 표시**를 우선한다.

### 표시 규칙
- 직계 조상은 중심 카드 또는 굵은 강조로 표시
- 해당 조상과 같은 아버지를 둔 형제를 옆에 함께 표시
- 형제 수는 최대 8명까지 표시
- 초과 시 “외 n명”으로 축약 가능
- 형제는 문중원 ID 순 또는 출생순 기준으로 정렬

### 모바일 표시 권장
- 초기 화면: 5~8대까지만 기본 표시
- 확장 보기: 최대 30대까지 펼침

## 7) 가계도 페이지 명세

가계도 페이지는 **D3.js 기반 세계도(세보) 트리**를 사용한다.

1차 구현은 SVG 방식으로 한다.  
Canvas는 2차 성능 검토 시 고려한다.

가계도 페이지 안에는 보조 탭 또는 세그먼트 버튼을 두어 다음 기능을 나눈다.

- 세보 트리
- 8촌 찾기
- 촌수 계산기

### 7-1. 세대 구간 선택
가계도 페이지에서는 전체 세대를 한 화면에 동일 방식으로 표시하지 않고, **세대 구간별로 다른 표시 방식**을 적용한다.

사용자는 다음 구간 중 하나를 선택하여 탐색할 수 있다.

- **선조 요약 구간(1세~24세)**
- **핵심 세보 트리 구간(25세~30세)**
- **31세 선택판**
- **후대 상세 트리 구간(32세 이후)**

각 구간의 표시 원칙은 다음과 같다.

- **1세~24세:** 전통 족보식 카드 및 분기 노드 중심의 요약 표시
- **25세~30세:** 4개 파(광택, 광진, 광룡, 광혁)를 포함한 본격 세보 트리 표시
- **31세:** 개별 인물을 선택할 수 있는 선택판 또는 카드 표
- **32세 이후:** 31세에서 선택된 인물을 root로 하는 하위 가계도 트리 표시

본인 확인이 완료된 경우에는, 선택된 인물을 기준으로 직계 조상에 해당하는 연결선(부계 직계 라인)을 일반 연결선보다 굵게 표시하여 시각적으로 강조한다.

### 7-2. 31세 선택판
31세는 별도 목록, 카드 영역 또는 선택판 형태로 표시한다.

사용자는 31세 인물 중 한 사람을 선택할 수 있으며, 선택된 인물은 **32세 이후 하위 가계도 트리의 root**가 된다.

이때, 선택된 인물 기준의 직계 조상 라인은 일반 가지와 구분되도록 굵은 선으로 강조 표시한다.

### 7-3. 트리 기능
가계도 페이지의 트리 화면에는 다음 기능을 포함한다.

- 확대
- 축소
- 초기화
- 접기
- 펼치기
- 특정 인물 선택 시 강조
- root 변경 후 re-render
- 중심 이동

특히 31세 이후의 하위 가계도에서는 확대/축소/중심 이동 기능이 중요하며, 사용자가 특정 인물의 후손 흐름을 탐색하기 쉽도록 한다.

### 7-4. 성별 시각 구분
성별에 따라 노드의 테두리 또는 색조를 다르게 표시한다.

권장 방식은 다음과 같다.

- 남성: 푸른 계열
- 여성: 분홍/보라 계열

너무 강한 대비보다는 차분한 색을 사용하여 전체 족보 화면의 안정감을 유지한다.

### 7-5. 촌수 계산기
촌수 계산기는 가계도 페이지 안의 보조 탭으로 배치한다.

구성 예:
- 사람 1 이름 검색
- 사람 2 이름 검색
- 동명이인 있을 경우 아버지 이름 등을 보고 후보 선택
- 계산 버튼
- 결과 표시

결과 표시:
- 두 사람의 촌수
- 공통 조상
- 관계 설명
- **두 사람의 공통 조상을 가운데에 두고, 양옆으로 두 사람을 가로선으로 연결하여 촌수 관계를 시각화한 그림 표시**

예를 들면, 공통 조상을 중심으로 왼쪽에 사람 1, 오른쪽에 사람 2를 두고, 각각의 세대 차이를 선으로 연결하여 가로형 촌수 관계도로 보여준다.

촌수 계산기는 하단 플로팅 메뉴의 독립 메뉴로 두지 않고, 가계도 페이지 내부 기능으로 포함한다.

### 7-6. 세보 트리 화면 구성
세보 트리는 한 화면에 모든 세대를 한꺼번에 펼치는 방식이 아니라, **족보식 상단 요약 + 분기 트리 + 선택형 하위 트리** 구조로 구성한다.

#### 상단 선조 요약 구간
1세부터 24세까지의 선조 구간은 전통 족보 느낌의 카드형 요약으로 표시한다.

- **1세~10세:** 시조부터 중시조까지 가로 카드형 연결
- **11세~20세:** 11세의 3개 분기 노드를 포함한 카드형 연결
- **21세~24세:** 세거지 분기와 후속 연결 요약

필요에 따라 이 구간은 다음과 같이 큰 카드 묶음으로 나누어 표시할 수 있다.

- **1세~10세**
- **11세~20세**
- **21세~24세**

이 구간은 트리보다 **가로 연결선 + 카드 + 세로 형제 표시** 방식이 적합하다.

#### 11세 분기 표시
11세에서 3개 분기 노드가 생긴다.

- 파조(평장사공파)
- 상서공파
- 기타 dd공파

내 조상 계통은 계속 아래로 연결하고, 나머지 2개 분기는 이름과 파명을 붙인 분기 노드로 표시하되, 이후 기록은 없는 상태로 남겨 둔다.

#### 21세 세거지 분기
21세에서 4개의 세거지 분기가 생긴다.

내 조상 계통 외의 나머지 3개 분기는 이후 자료가 없더라도, 족보 방식으로 세거지에 따른 분기 노드로 표시해 둔다.

#### 25세 이후 핵심 파 분기
25세에서 다음 4개 파가 중요하게 분기된다.

- 광택
- 광진
- 광룡
- 광혁

이 4개 분기는 이후 후손 자료가 크게 늘어날 것으로 보므로, **화면상 동일한 크기의 공간을 미리 확보**한다.

현재는 광혁파의 자료가 가장 많이 정리되어 있으나, 향후 광택·광진·광룡도 비슷한 규모로 확장된다는 가정하에 여유 있는 트리 공간을 확보한다.

#### 25세~30세 트리
25세부터 30세까지는 4개 파를 포함한 본격적인 세보 트리를 구성한다.

- 25세를 4개 파의 큰 분기 루트로 표시
- 26세~30세는 각 파별 하위 가지 전개
- 현재 자료가 있는 광혁파는 실제 노드를 적극 반영
- 나머지 3개 파는 최소 자료만 반영하되, 향후 확장 공간을 확보

#### 31세 선택판
31세는 17명을 한 표 또는 카드 영역으로 모아 표시한다.

사용자가 31세 인물 중 한 사람을 선택하면, 해당 인물을 root로 하여 **32세 이후의 하위 가계도**를 별도 트리로 다시 그린다.

#### 32세 이후 하위 트리
32세 이후는 인원수가 급격히 증가하므로, 31세 선택 후 하위 가계도를 표시하는 방식으로 구성한다.

이 하위 트리에는 다음 기능을 포함한다.

- 확대
- 축소
- 중심 이동
- 초기화
- 선택 인물 기준 재정렬

이 구조는 향후 전체 문중원의 기록이 1000~1500명 규모로 확장되더라도 현실적으로 대응 가능한 방식으로 본다.

## 8) 8촌 이내 친족 찾기 명세

기준 인물 ID를 기준으로 위로 4대, 즉 고조부까지 추적한다.  
그 후 고조부를 root로 하여 하위 후손 집합을 구성하고, 그 안에서 8촌 범위에 해당하는 인물을 필터링한다.

### 기본 흐름
1. 본인 찾기
2. 부 → 조부 → 증조부 → 고조부 추적
3. 고조부 하위 후손 전체를 트리 구조로 구성
4. 촌수 또는 세대거리 기준으로 8촌 범위를 판별
5. 결과를 **하나의 인터랙티브 트리**로 표시

### 표시 방식
- 결과는 카드 목록이 아니라 **단일 트리 화면**으로 표시한다
- 사용자는 확대, 축소, 드래그 이동, 초기화를 통해 트리를 탐색할 수 있다
- 본인 노드는 별도로 강조 표시한다
- 본인의 직계 조상 라인은 굵은 선으로 표시한다
- 8촌 이내 인물은 선명하게 표시하고, 그 외 인물은 흐리게 표시하거나 숨김 처리할 수 있다

### 화면 원칙
- 전체 인원이 대략 100명 이내일 경우 하나의 트리로 렌더링한다
- 초기 화면은 전체 갈래가 보이도록 자동 맞춤한다
- 필요시 “8촌만 보기 / 전체 갈래 보기” 전환 기능을 제공할 수 있다

## 10) 아천문중 페이지
아천문중 페이지는 문중의 역사와 선조 관련 자료를 함께 볼 수 있는 페이지로 구성한다.

### 기본 구성
아천문중 페이지는 다음과 같은 보조 탭으로 나눈다.

- **가문의 연표와 선조**
- **대동보와 족보이야기**
- **정관 / 문중재산**
- **문중원 투표**

### 탭 내용 원칙
위 각 탭의 세부 내용과 구성은 앞선 통합 2 최종본의 관련 내용을 따른다.

즉,
- **가문의 연표와 선조**: 기존 연표, 선조 관련 자료, 인물 정보
- **대동보와 족보이야기**: 대동보 PDF 자료, 족보 이야기, 문중 전승
- **정관 / 문중재산**: 문중 운영 관련 자료, 정관, 재산 관련 안내나 문서
- **문중원 투표**: 기존 투표 기능 구성

### 문중원 투표
족보 앱에는 문중원 대상의 간단한 투표 기능을 부착할 수 있다.

투표 기능은 **한 사람이 한 번만 참여 가능**하도록 설계한다.

#### 기본 방식
가장 단순한 1차 방식으로, 사용자는 다음만 입력한다.

- 이름
- 투표 항목 선택

#### 중복 방지 방식
- 동일 안건에서 **같은 이름이 이미 기록되어 있으면 재투표를 막는다**
- 이미 투표한 경우 “이미 참여하셨습니다”와 같은 안내 문구를 표시한다

#### 저장 시트
별도 `Vote` 또는 `VoteResponse` 시트를 둔다.

예시 컬럼:
- voteId
- agendaId
- voterName
- selectedOption
- votedAt
- isValid

#### 동작 흐름(권장)
1. 사용자는 투표 화면에서 **이름**과 **선택 항목**을 입력/선택한다.
2. 앱은 먼저 `voteCheck`로 **중복 투표 여부**를 확인한다.
3. 중복이 아니면 `voteSubmit`으로 투표를 제출한다.
4. Apps Script는 `VoteResponse` 시트에 한 줄을 추가하고, 성공 메시지를 반환한다.
5. 앱은 성공 메시지를 표시하고, 필요 시 `vote`(안건/득표수) 또는 `voteResult`를 재조회하여 화면을 갱신한다.

#### 중복 판정 규칙(1차)
- **동일 안건(`agendaId`) + 동일 이름(`voterName`)** 조합이 이미 존재하면 중복으로 간주한다.
- 1차 구현은 운영 편의상 **이름 기반**으로만 중복을 막는다(강한 인증/실명 검증은 하지 않음).

#### Apps Script API 명세(예시)

##### 1) 투표 안건 조회(표시용)
`?action=vote`

응답 예:
```json
{
  "agendaId": 1,
  "title": "2026년 문중 행사 장소 투표",
  "options": ["의성", "대구", "온라인"],
  "votes": [12, 7, 3]
}
```

##### 2) 중복 확인
`?action=voteCheck&agendaId=1&voterName=홍길동`

응답 예:
```json
{
  "agendaId": 1,
  "voterName": "홍길동",
  "isDuplicate": true,
  "message": "이미 참여하셨습니다."
}
```

##### 3) 투표 제출
`?action=voteSubmit`

전달값 예:
- agendaId
- voterName
- selectedOption

권장: `POST`(폼 URL 인코딩) 또는 `GET` 모두 가능하나, **가능하면 POST**를 권장한다.

응답 예:
```json
{
  "success": true,
  "voteId": "V-20260413-0001",
  "message": "투표가 저장되었습니다."
}
```

실패(중복) 예:
```json
{
  "success": false,
  "isDuplicate": true,
  "message": "이미 참여하셨습니다."
}
```

#### 비고
이 방식은 가장 단순하고 현실적이지만, **동명이인 구분이나 강한 인증 기능은 고려하지 않는다.**  
보안보다 사용 편의와 간단한 운영을 우선하는 1차 방식으로 본다.
